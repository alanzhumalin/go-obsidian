### Вводная часть

Сервер не может давать данные всем подряд, поэтому клиент сначала должен представиться. Стоит различать ***Авторизацию, Аутентификацию и Идентификацию***.

***Идентификация*** - это процедура распознавания субъекта по его идентификатору (проще говоря, это определение имени, логина или номера)

***Аутентификация*** - процедура проверки подлинности (пользователя проверяют с помощью пароля, письмо проверяют по электронной подписи и т.д.)

***Авторизация*** - предоставление доступа к какому-либо ресурсу (например, к электронной почте)

## Виды авторизации
### Сессии
- Хранится в ***cookies***
	- Сервер кладёт ***id*** сессии в ***cookies***, которые отправляет клиенту при успешной авторизации
- При каждом запросе отправляется на сервер
- Server-side (в основном)
- Не для публичного ***API***.

### Токены

### 🔐 **JWT (JSON Web Token) в Go**

JWT (**JSON Web Token**) — это компактный токен, который используется для **аутентификации** и **авторизации** пользователей (удобный способ аутентификации, особенно в REST API)

В Go JWT часто применяется для:  
-  Авторизации API-запросов (REST/gRPC).  
-  Хранения данных пользователя без серверного состояния (stateless).  
- Работа с OAuth и микросервисами.

---

- Публичное ***API***
- Микросервисная Архитектура
- Обычно отправляются в заголовках запроса
- В браузере хранятся либо в ***DOM-хранилищах(sessionStorage/localStorage)***, либо в тех же ***cookies***.
- Короткое время жизни (токен для доступа к данным)

## **Как устроен JWT?**

JWT состоит из **трёх частей**, разделённых точками (`.`):

```
header.payload.signature
```

Пример JWT:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOjEsImV4cCI6MTY5NTM4NjAwMH0.5Jtk5NRHfp9BQkd...
```

Разберём каждую часть:

| Часть         | Описание                                                      |
| ------------- | ------------------------------------------------------------- |
| **Header**    | Заголовок: алгоритм подписи (`HS256`, `RS256` и др.)          |
| **Payload**   | Полезная нагрузка: данные (например, `userID`, `role`, `exp`) |
| **Signature** | Подпись: защита от подделки (шифруется секретным ключом)      |

### Почему используется 2 токена?

- ***Access Token***
	- многоразовый
	- короткоживущий (в среднем от 15 минут до нескольких часов)
	- Нужен, чтобы получать доступ к системе

- ***Refresh Token***
	- одноразовый
	- долгоживущий (от нескольких недель до месяца)
	- нужен, чтобы обновлять ***Access Token***
