`panic` — это механизм аварийного прерывания нормального потока.  
При `panic` Go разворачивает стек текущей goroutine и выполняет `defer` в порядке `LIFO`.  
Если в одном из этих `defer` вызывается `recover()`, паника перехватывается (в этой же goroutine), и процесс может не упасть.

Ключевой нюанс: `recover` работает только внутри `defer` и только для текущей goroutine.


`panic` в Go — это аварийное завершение текущего потока выполнения (goroutine), когда нормальная обработка ошибки уже не используется.

**Что происходит при panic**
1. Останавливается обычный путь выполнения.
2. Go начинает “разворачивать стек” этой goroutine вверх.
3. Выполняет `defer`-ы в порядке `LIFO` (последний `defer` выполняется первым).
4. Если в одном из `defer` вызван `recover()`, паника перехватывается, и выполнение продолжается после `defer`.
5. Если `recover` не сработал, процесс падает с stack trace.

**Когда panic возникает**
1. Явный вызов `panic(...)`.
2. Runtime panic: `index out of range`, `nil pointer dereference`, `divide by zero` и т.д.
3. Некоторые фатальные ошибки runtime (например, гонки записи/чтения в map) завершают программу как `fatal error`; это не нормальный сценарий для `recover`.

**Короткий ответ на интервью (готовая формулировка)**
`panic` — механизм аварийного завершения в Go. Он не заменяет `error`: для ожидаемых ошибок возвращаем `error`, для нарушенных инвариантов и “невозможных” состояний допустим `panic`. При панике стек разворачивается, `defer` выполняются в LIFO, а `recover` может перехватить панику только внутри `defer` и только в той же goroutine.

**Самые частые вопросы на интервью + ответы**
1. `panic` vs `error`?
Ответ: `error` для ожидаемых бизнес-ошибок, `panic` для критических/невозможных состояний.
2. Где работает `recover`?
Ответ: только внутри `defer` и только в той же goroutine.
3. Можно ли поймать panic из другой goroutine?
Ответ: нет.
4. Выполняются ли `defer` при panic?
Ответ: да, в обратном порядке.
5. Выполняются ли `defer` при `os.Exit`?
Ответ: нет.
6. Что делает `log.Panic`?
Ответ: логирует и затем вызывает `panic`.
7. Нормально ли паниковать в библиотеке?
Ответ: обычно нет, лучше возвращать `error`; panic только при нарушении внутренних инвариантов.
8. Что если panic случился внутри `defer` во время разворачивания?
Ответ: обычно приводит к аварийному завершению (double panic), ситуация плохая.
9. Можно ли использовать panic как control flow?
Ответ: не рекомендуется.
10. Нужно ли `recover` делать в каждом месте?
Ответ: обычно ставят “защитный” `recover` на границе goroutine/worker/HTTP middleware.

**Подводные камни (“приколы”)**
1. `recover` вне `defer` не работает.
2. `recover` в одной goroutine не ловит панику другой.
3. Паника может “убить” весь процесс, если не перехватить на границе.
4. `defer` выполняется даже при panic, но не при `os.Exit`.
5. Часто на интервью спрашивают, почему `panic` не должен использоваться вместо `error`.

**Мини-шаблон безопасной границы**
```go
defer func() {
    if r := recover(); r != nil {
        // лог + метрика + корректный ответ/завершение worker
    }
}()
```
