***HTTP*** — протокол прикладного уровня, использующий TCP для передачи данных. Основные особенности:
- Текстовый формат — запросы и ответы передаются в виде текста (например, `GET / HTTP/1.1`).
    - Без сохранения состояния — каждый запрос обрабатывается независимо.
    - Примеры: загрузка веб-страниц, API-запросы

***TCP*** — протокол транспортного уровня, обеспечивающий надёжную передачу данных:
- Установка соединения — перед отправкой данных создаётся сессия (тройное рукопожатие).
- Потоковая передача — данные делятся на сегменты с номерами, гарантирующими порядок доставки[

***UDP*** — протокол транспортного уровня для быстрой передачи без гарантий:

- Дейтаграммы — пакеты отправляются независимо, без подтверждения получения.
- Низкая задержка — подходит для потокового видео, голосовых вызовов, онлайн-игр
- Примеры: Skype, Discord, DNS-запросы[
    
Сравнение:

|Характеристика|TCP|UDP|
|---|---|---|
|**Надёжность**|Гарантирует доставку|Потери пакетов допустимы|
|**Скорость**|Медленнее из-за проверок|Быстрее|
|**Примеры**|HTTP, файлы, почта|Видео, голос, игры|

HTTP всегда использует TCP, так как требует точной доставки данных. UDP применяется там, где важна скорость, а потери пакетов не критичны

## Пример использования HTTP и установка соединения

## Пример запроса

Пользователь вводит в браузер:  
`http://example.com  `

HTTP-запрос (текстовый формат):
```
GET / HTTP/1.1   Host: example.com
HTTP-ответ сервера:
HTTP/1.1 200 OK 
Content-Type: text/html 
<!DOCTYPE html> 
<html>...</html>
```

## Установка соединения

1. Резолвинг домена:
    - Клиент (браузер) преобразует example.com в IP-адрес через DNS.
        
2. Создание TCP-соединения:
    - Клиент устанавливает TCP-соединение с сервером по порту 80 (по умолчанию для HTTP).
    - Тройное рукопожатие:
        - SYN → SYN-ACK → ACK — подтверждение готовности к передаче данных
        
3. Отправка HTTP-запроса:
    - После установления соединения клиент передаёт запрос (например, `GET /`).
    - Обязательные элементы: метод, URI, версия HTTP, заголовок `Host
    
4. Обработка и ответ сервера:
    - Сервер возвращает статус (`200 OK`), заголовки и тело ответа (HTML, изображение и т.д.)

5. Закрытие соединения:
    - В HTTP/1.1 соединение может оставаться открытым для последующих запросов (`keep-alive`).

## Ключевые особенности

- Без сохранения состояния: каждый запрос обрабатывается независимо
- Прокси-серверы: могут кэшировать данные или шифровать трафик (например, для HTTPS)
- HTTP vs HTTPS: HTTPS добавляет шифрование (TLS/SSL), но базовая логика соединения остаётся такой же[
