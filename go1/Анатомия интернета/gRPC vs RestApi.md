
###  **Различия между gRPC и REST API в плане передачи данных**

Один из **ключевых отличий** между **gRPC** и **REST API** — это формат данных.

- В **REST API** данные обычно передаются в **JSON**
- В **gRPC** используется **протокол Protobuf (Protocol Buffers)**, который **строго типизирован**.


---

## 🔍 **REST API: передача данных в JSON**

###  **Как JSON работает в REST API?**

- Данные передаются как **текст (строка)** в формате JSON.
- JSON **не имеет строгой типизации** — все данные хранятся в виде `ключ: значение`.
- Сервер **не может заранее знать** точный тип данных в теле запроса.

### **Пример JSON-запроса в REST API**

```json
{
    "id": 1,
    "name": "Alice",
    "age": 25
}
```

⚠️ **Проблема:** JSON **не гарантирует** строгую типизацию. Клиент может отправить:

```json
{
    "id": "one",   // Ошибка: id строкой вместо числа
    "name": 123,   // Ошибка: name числом вместо строки
    "age": "25"    // Ошибка: age строкой вместо числа
}
```

В таких случаях серверу приходится **дополнительно проверять** тип данных.

---

## **gRPC: передача данных в Protobuf**

###  **Как работает Protobuf в gRPC?**

- Данные передаются в бинарном формате, а не в виде текста (меньше трафика).
- **Протокол строго типизирован** — структура данных описывается в `.proto` файлах.
- Ошибки из-за неправильных типов **невозможны**, так как схема фиксирована.

###  **Пример `.proto` файла для gRPC**

```proto
syntax = "proto3";

message User {
    int32 id = 1;
    string name = 2;
    int32 age = 3;
}
```

- `int32 id = 1;` → `id` **обязательно** должен быть `int32`.
- `string name = 2;` → `name` **только строка**.
- `int32 age = 3;` → `age` **только число**.

 **Если клиент отправит не тот тип данных — будет ошибка на уровне сериализации!**  
 **Сервер заранее знает, какие данные он получит.**

---

## **Основные отличия gRPC и REST API в плане данных**

|Характеристика|REST API (JSON)|gRPC (Protobuf)|
|---|---|---|
|**Формат данных**|JSON (текст)|Protobuf (бинарный)|
|**Размер данных**|Большой (из-за текста)|Маленький (сжатый бинарный)|
|**Типизация**|Нет строгой типизации|**Жёсткая типизация** (через `.proto`)|
|**Скорость передачи**|Медленнее (из-за парсинга JSON)|Быстрее (из-за бинарного формата)|
|**Ошибка в типе данных**|Возможна (сервер должен проверять)|**Невозможна** (компилятор проверяет)|

---

## **Почему строгая типизация в gRPC — это плюс?**

✅ **Отсутствие ошибок с типами** → Клиент не может отправить данные неправильного типа.  
✅ **Производительность выше** → JSON нужно **парсить**, а Protobuf сразу раскодируется.  
✅ **Экономия трафика** → Бинарный формат в 5-10 раз компактнее JSON.

---

##  **Вывод**

1️⃣ **REST API (JSON)** удобен для работы в браузере, но требует дополнительных проверок типов.  
2️⃣ **gRPC (Protobuf)** гарантирует строгую типизацию, быстрее передаёт данные и снижает трафик.

Если тебе нужно **API, которое должно быть строгим, быстрым и надёжным** → выбирай **gRPC**.  
Если API должно работать **с веб-приложениями и быть простым** → **REST API (JSON)**.
