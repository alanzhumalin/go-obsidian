# REST API

Это архитектурный стиль(шаблон), набор правил, который описывает то, как наиболее эффективно использовать HTTP и строить свою API так, чтобы ей было удобно пользоваться, чтобы она выдерживала нагрузки, легко масштабировалась и т.д.

> Обмен данными через JSON/XML
### REST API Концепции

- ***Модель взаимодействия:*** клиент - сервер

![[Pasted image 20250313190650.png]]

- ***Система может быть многоуровневой***

![[Pasted image 20250313190700.png]]

- ***Stateless Отсутствие состояния.*** Говорит нам о том, что сервер не должен обладать каким-либо состоянием

![[Pasted image 20250313190711.png]]
> Каждый раз клиент и сервер общаются как в первый раз. И для того чтобы сервер идентифицировал клиента, понял, что ему необходимо сделать - клиент должен отправить вновь всю информацию, необходимую для выполнения запроса

- ***единообразный унифицированный интерфейс***

![[Pasted image 20250313190728.png]]

> [!Важно]
> Также должна соблюдаться идемпотентность.
> Идемпотентность - это свойство, которое означает, что повторный идентичный запрос, сделанный один или несколько раз подряд, имеет один и тот же эффект, не изменяющий состояние сервера. Корректно реализованные методы GET, PUT и DELETE идемпотентны, но не метод POST

- ***кеширование***

![[Pasted image 20250313190742.png]]

> (Вместо того, чтобы постоянно идти в базу данных и вытаскивать данные, мы сохраняем информацию, которая редко обновляется в КЭШ)

- ***формат обмена данными:***
	- JSON - чаще всего
	- XML
