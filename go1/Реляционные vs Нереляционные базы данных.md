# Реляционные базы данных

Таблица представляет собой плоскую двумерную сетку, которая содержит определённый тип или типы структурированных данных, например фильмы,  группы крови, маршруты движения и т. д.

==База данных табличного вида==, в которой жёсткие связи 
Каждая строка содержит фиксированное кол-во полей и все они должны быть одинаковые 
# Нереляционные базы данных

Они отличаются от реляционных тем, что в них для хранения используется не система из строк и столбцов, а применяется модель, которая оптимизирована для хранения определённого типа содержимого. Например, данные могут храниться в виде документов JSON, графов, а также ключей-значений.

Причём здесь важным моментом является то, что ==в отличие от реляционной модели хранения==, где данные одного объекта могут быть распределены по разным таблицам, ==здесь они хранятся в одном документе, и этот тип хранения не требует, чтобы все документы имели одинаковую структуру==, благодаря чему и достигается достаточно большая гибкость.

### Реляционные базы данных (SQL)  
Хранят данные в таблицах со строками и столбцами, где каждая строка — отдельная запись, а столбцы — атрибуты с фиксированными типами данных.  
Особенности:

- Строгая схема: структура таблиц и связи между ними задаются заранее
- ACID-транзакции: гарантируют целостность данных (атомарность, согласованность и т.д.)
- SQL: используется для запросов, сложных операций и аналитики
    Примеры: MySQL, PostgreSQL, Oracle
    

### Нереляционные базы данных (NoSQL)  
Хранят данные в гибких форматах: документы, ключ-значение, графы
Особенности:

- Гибкая схема: структура данных может меняться без предварительного определения

- Масштабируемость: оптимизированы для больших объёмов данных и высокой нагрузки

- Простые запросы: часто используют специализированные API или языки 
    Примеры: MongoDB, Redis, Cassandra
    

Когда выбрать:

- Реляционные — для структурированных данных, сложных запросов, банковских систем
    
- Нереляционные — для неструктурированных данных, социальных сетей, IoT

### 📌 **Пример использования реляционной и нереляционной БД**

#### **1️⃣ Реляционная база (SQL, PostgreSQL)**

**Применение:** Интернет-магазин (хранение пользователей и заказов).

📊 **Таблица `users`**

|id|name|email|
|---|---|---|
|1|Alice|[alice@mail.com](mailto:alice@mail.com)|
|2|Bob|[bob@mail.com](mailto:bob@mail.com)|

📊 **Таблица `orders` (связь с `users`)**

|id|user_id|total|
|---|---|---|
|1|1|100|
|2|2|200|

🔹 **SQL-запрос:**

```sql
SELECT users.name, orders.total 
FROM users 
JOIN orders ON users.id = orders.user_id 
WHERE users.id = 1;
```

📌 **Выдаст:** `"Alice", 100`

---

### 📌 **Пример, где реляционная БД** не подходит, а NoSQL — идеален

#### **📍 Сценарий: Логирование событий в реальном времени (лог-система)**

**Почему SQL ?**

- Миллионы записей в секунду → **SQL не выдержит нагрузку**.
- Данные неструктурированы → могут содержать **разные поля** (IP, user-agent, ошибки).
- Быстрая вставка и поиск → SQL **медленно индексирует логи**.

### 🚀 Пример: Каталог товаров с динамическими атрибутами

**Сценарий:**  
Ты создаёшь магазин, где товары из разных категорий имеют **совсем разные атрибуты**.

- **Футболка:** размер, цвет, материал
- **Ноутбук:** процессор, оперативная память, объём SSD, диагональ экрана
- **Книга:** автор, жанр, год издания

**Проблема в SQL:**  
Чтобы вместить все возможные поля, нужно либо:

- Добавлять много колонок (многие из которых будут пустыми),
- Либо создавать сложную схему с дополнительными таблицами для динамических атрибутов.

**Решение в NoSQL (например, MongoDB):**  
Каждый товар хранится как **JSON-документ** с набором полей, специфичных для его категории.

```json
// Пример документа для футболки
{
    "product_id": 101,
    "category": "clothing",
    "name": "Футболка",
    "attributes": {
        "size": "L",
        "color": "blue",
        "material": "cotton"
    }
}
```

```json
// Пример документа для ноутбука
{
    "product_id": 202,
    "category": "electronics",
    "name": "Ноутбук",
    "attributes": {
        "processor": "Intel i7",
        "ram": "16GB",
        "storage": "512GB SSD",
        "screen_size": "15.6 inches"
    }
}
```

**Почему только NoSQL:**

- Нет необходимости фиксировать схему заранее.
- Каждый документ может иметь уникальную структуру, что идеально подходит для товаров с различными атрибутами.

Таким образом, для каталога товаров с динамическими и разными атрибутами реляционная база будет слишком громоздкой, а NoSQL позволяет хранить данные естественно и гибко.


### 🏆 **Плюсы и минусы реляционных (SQL) и нереляционных (NoSQL) баз данных**

---

## 📌 **Реляционные базы данных (SQL)**

✔ **Хранят данные в таблицах**, используют **жёсткую схему** и **SQL-запросы**
✔ Примеры: PostgreSQL, MySQL, SQLite, MS SQL Server.

### ✅ **Плюсы SQL**

✔ **Строгая структура** → данные организованы, меньше ошибок.  
✔ **Связи (JOIN)** → легко объединять данные из разных таблиц.  
✔ **Поддержка ACID** → данные всегда **консистентны** (без потерь и дубликатов).  
✔ **Хорош для сложных запросов** → удобные SQL-запросы с фильтрацией, сортировкой и агрегацией.  
✔ **Широко используется** → много документации, инструментов, специалистов.

### ❌ **Минусы SQL**

⛔ **Медленнее на больших объёмах данных** → сложные `JOIN`-запросы могут тормозить.  
⛔ **Жёсткая схема** → изменения структуры (например, добавление новых полей) требует миграций.  
⛔ **Сложно масштабировать** → горизонтальное масштабирование (разделение на несколько серверов) сложное.

---

## 📌 **Нереляционные базы данных (NoSQL)**

✔ Хранят данные в **документах (JSON), ключ-значение, графах или колонках**.  
✔ Примеры: MongoDB (документы), Redis (ключ-значение), Cassandra (колонки), Neo4j (графы).

### ✅ **Плюсы NoSQL**

✔ **Гибкость** → можно хранить данные **без фиксированной схемы**.  
✔ **Высокая скорость** → быстрее обрабатывает **неструктурированные и большие данные**.  
✔ **Масштабируемость** → легко распределять нагрузку на **несколько серверов (горизонтальное масштабирование)**.  
✔ **Подходит для Big Data** → идеально для логов, аналитики, соцсетей.

### ❌ **Минусы NoSQL**

⛔ **Нет SQL и JOIN** → сложно делать сложные запросы между связанными данными.  
⛔ **Нет строгих ограничений на данные** → могут появляться ошибки из-за отсутствия схемы.  
⛔ **Не всегда поддерживает ACID** → в некоторых NoSQL-базах возможны **неконсистентные данные**.

---

## 🔥 **Когда использовать SQL и NoSQL?**

| Ситуация                                    | Что выбрать?                             |
| ------------------------------------------- | ---------------------------------------- |
| **Банковские транзакции** 💳                | SQL (надёжность, ACID)                   |
| **Каталог товаров с разными атрибутами** 🛍 | NoSQL (гибкость)                         |
| **Соцсети, мессенджеры** 📱                 | NoSQL (быстрая работа, масштабируемость) |
| **Хранение логов и аналитики** 📊           | NoSQL (скорость, большие данные)         |
| **CRM, учёт заказов** 📋                    | SQL (структурированные данные)           |

💡 **Вывод:**

- **SQL → когда важны структура, связи и надёжность.**
- **NoSQL → когда нужна гибкость, скорость и масштабируемость.**
